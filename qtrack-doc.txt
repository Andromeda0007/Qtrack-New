• controlled user access

Question 3
How is Role-Based Access Control (RBAC) implemented in QTrack?
Role-Based Access Control (RBAC) is the core security and workflow control mechanism of the entire QTrack system.
The purpose of RBAC is to ensure:
• only authorized users perform certain actions
• workflow steps follow the correct order
• accidental or malicious misuse is prevented
• system behavior remains predictable and scalable
Without RBAC, users could perform actions outside their responsibilities (for example a warehouse worker approving material quality), which would break the entire operational integrity of the warehouse system.
________________________________________
1. Why RBAC is Necessary in QTrack
The warehouse workflow contains multiple departments:
• Warehouse
• Quality Control (QC)
• Quality Assurance (QA)
• Production
• Purchase
Each department has specific responsibilities.
Example:
Role	Responsibility
Warehouse	Receive materials
QC Executive	Test materials
QC Head	Approve or reject materials
QA Executive	Inspect finished goods
Production	Create finished goods
Purchase	View stock
Because of this separation, the system must enforce strict permission boundaries.
________________________________________
2. Roles Defined in the System
Based on the business requirements, QTrack contains 8 main roles.
1. Super Admin
System administrator.
Responsibilities:
• create users
• assign roles
• deactivate accounts
• view audit logs
Super Admin does not participate in warehouse operations.
________________________________________
2. Warehouse User
Operational warehouse staff.
Permissions:
• create GRN (Goods Receipt Note)
• update product location
• issue material to production
• receive finished goods
• dispatch finished goods
Cannot approve or reject materials.
________________________________________
3. Warehouse Head
Warehouse supervisor.
Permissions include everything Warehouse User can do plus:
• revise incorrect GRN entries
• reprint quarantine labels
• approve warehouse corrections
________________________________________
4. QC Executive
Quality Control testing staff.
Permissions:
• generate A.R. number
• withdraw samples
• mark materials as UNDER TEST
Cannot approve or reject materials.
________________________________________
5. QC Head
Quality authority.
Permissions:
• approve materials
• reject materials
• set retesting date
• approve grade transfer (IP → BP / USP)
________________________________________
6. QA Executive
Quality Assurance inspection staff.
Responsibilities:
• verify finished goods quantity
• perform physical quality check
________________________________________
7. QA Head
Final QA authority.
Permissions:
• approve finished goods
• reject finished goods
• authorize correction of QA entries
________________________________________
8. Production User
Production department.
Permissions:
• submit finished goods to warehouse
• generate shipper labels
________________________________________
9. Purchase Department
Read-only role.
Permissions:
• view stock
• view reports
Cannot modify anything.
________________________________________
3. RBAC Design Philosophy
We avoid writing code like this:
if user.role == "QC_HEAD":
    allow_approval
Because that becomes messy as the system grows.
Instead we use permission mapping.
Structure:
Users → Roles → Permissions
________________________________________

6. Permission Enforcement in Backend
When an API request arrives:
Example request:
POST /approve-material
Backend performs checks:
1. Verify JWT token
2. Identify user
3. Identify user role
4. Check permission
If permission exists:
ALLOW ACTION
Otherwise:
403 Forbidden
________________________________________
7. Example RBAC Flow
Example scenario:
A warehouse user attempts to approve material.
User Role: WAREHOUSE_USER
Action: APPROVE_MATERIAL

Does WAREHOUSE_USER have APPROVE_MATERIAL permission?
Answer:
No
Response:
403 Forbidden
Action blocked.
________________________________________
8. Workflow Protection Through RBAC
RBAC also protects process order.
Example:
Material lifecycle:
GRN
↓
QUARANTINE
↓
UNDER TEST
↓
APPROVED / REJECTED
Permissions enforce this flow.
Example:
Role	Allowed Action
Warehouse	create GRN
QC Executive	set UNDER TEST
QC Head	approve/reject
This ensures the workflow cannot be bypassed.
________________________________________
9. UI Permission Control
The mobile app also respects roles.
Example:

The UI hides actions users cannot perform.
But backend enforcement is still mandatory.
Never rely only on frontend restrictions.

If a user's role changes:
Example:

We update the role_id in the user table.
We do not create a new user.
Why?
Because the audit logs remain linked to the same identity.
Example:

All actions are logged.

This ensures accountability.

12. Advantages of This RBAC Design
This RBAC model provides:
Security
Unauthorized actions are blocked.
Scalability
New roles can be added easily.
Maintainability
Permissions are managed in the database.
Compliance
Audit logs record all actions.
________________________________________

14. Future Expansion
RBAC allows easy expansion.
Example:

Done
No code rewrite required.
________________________________________
Summary
The RBAC system works as:

Security checks occur at:
• API level

• UI level
This design ensures the warehouse system operates with strict control and accountability.

Question 4
How are users created, managed, deactivated, and updated in QTrack?
User lifecycle management is an important part of the system because QTrack is a closed enterprise ecosystem. Only authorized employees should be able to access the system, and every user action must remain traceable for auditing and compliance.
Therefore, the user management system must support:
• controlled user creation
• role assignment
• role updates
• account deactivation
• audit tracking of user actions
• password resets
• secure onboarding
The system is designed so that users cannot self-register. All user management operations are performed by the Super Admin.
________________________________________
1. User Creation Process
Users are created only by the Super Admin.
This ensures that:
• only trusted individuals enter the system
• roles are correctly assigned
• permissions are controlled
The creation process follows this flow:
Super Admin → Create User → Assign Role → System Generates Temporary Password → Email Sent → User Logs In
________________________________________
2. User Creation Interface
Inside the admin dashboard, the Super Admin will see a form to create users.
Required fields:
Full Name
Username
Email Address
Role
Phone (optional)
Account Status
Example:
Field	Value

Username	ankit.qc
Email	ankit@company.com
Role	QC Executive
The system then generates a temporary password automatically.
________________________________________
3. Temporary Password Generation
The password should not be created manually by the admin.
Instead, the system generates a random password.
Example:
Gx91@kaL8P
Why?
• prevents weak passwords
• ensures admin does not know the user's password
• improves security
The password is then hashed using bcrypt before storing.
Example stored value:

________________________________________
4. Sending Credentials via Email
After user creation, an email is sent to the user.

In regulated industries such as pharmaceuticals or chemicals, the same material may have different quality grades depending on testing standards.
Examples of common grade standards include:
Standard	Meaning
IP	Indian Pharmacopoeia
BP	British Pharmacopoeia
USP	United States Pharmacopeia
A material may arrive labeled as IP grade, but after laboratory testing it may qualify as BP or USP grade.

This process is called Code-to-Code Grade Transfer.

Imagine this scenario:
A shipment arrives labeled:

Convert material code from IP → USP

• audit history would be lost
• regulatory traceability would break
• inventory tracking would become inaccurate
Instead, the system performs a controlled transfer between material codes.

↓

↓

Question 12
How is inventory stored, updated, and controlled inside the database?
Inventory management is the core responsibility of the QTrack system. Everything else — QC testing, production, QA approval — ultimately depends on accurate stock tracking.
A poorly designed inventory system leads to:
•	incorrect stock levels
•	negative inventory
•	traceability failures
•	production interruptions
Therefore, QTrack uses a movement-based inventory model, which is the same approach used in large ERP systems (SAP, Oracle ERP, etc.).

Question 15
How will authentication and security work in QTrack (login, JWT, MFA, password reset)?
Security is one of the most important aspects of the QTrack system. The client specifically requested that:
•	outsiders must not be able to enter the ecosystem
•	accounts must only be created internally
•	the system must remain secure without complex network setups
Therefore, authentication in QTrack follows a controlled-entry model.
Only users created by the Super Admin can access the system.
________________________________________
1. Core Authentication Principles
The authentication design follows these rules:
No public signup
Accounts created by Super Admin
Username + Password login
Email used for verification and recovery
Mandatory password reset on first login
Role-based access control
Optional MFA for stronger security
This ensures the system remains closed and secure.

The login screen will contain only two fields:
Field	Purpose
Username	User identification
Password	Authentication
Example UI:
QTrack Login

Username: __________
Password: __________

[ Login ]
[ Forgot Password ]
Important:
No "Sign Up" option exists.
This prevents outsiders from registering.

When creating a new user, the admin must provide:
Field	Example

SMTP is required for:
•	password reset emails
•	new user credentials
•	system alerts
Example SMTP providers:
Provider	Notes
Amazon SES	recommended for production
SendGrid	easy integration
Gmail SMTP	good for development

smtplib
fastapi-mail

MAIL_SERVER = "smtp.gmail.com"

MAIL_TLS = True

6. First Login Password Reset
When a user logs in for the first time, the system checks:
is_first_login = true
If true:
Redirect → Password Reset Screen
The user cannot skip this step.
Example screen:
Create New Password

New Password: __________
Confirm Password: ______

[ Update Password ]
After update:
is_first_login = false

7. Password Storage Security
Passwords are never stored as plain text.
Instead they are hashed.
Recommended hashing algorithm:
bcrypt
Example:
from passlib.hash import bcrypt

hashed_password = bcrypt.hash("user_password")
Stored value example:
$2b$12$1S6w3G...
Even administrators cannot see user passwords.
________________________________________
8. Login Authentication Process
When a user logs in:
User enters username and password
↓
Backend retrieves stored password hash
↓
Password compared using bcrypt
↓
If valid → user authenticated
If invalid:
Login rejected
________________________________________
9. JWT Authentication
After successful login, the system generates a JWT token.
JWT = JSON Web Token.
This token allows the user to stay logged in.
Example token payload:
{
  "user_id": 12,
  "username": "raj.wh",
  "role": "warehouse_user",
  "exp": 1712352352
}
The token is sent to the frontend.
________________________________________
10. Token Usage
Every API request includes the token.
Example request header:
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Backend verifies:
•	token signature
•	token expiry
If valid → request allowed.
________________________________________
11. Token Expiration
JWT tokens should expire automatically.
Recommended settings:
Token	Expiry
Access Token	24 hours
Refresh Token	7 days
If token expires, the user must log in again.
________________________________________
12. Role-Based Access Control
After authentication, authorization checks the user's role.
Example:
Role	Allowed Action
Warehouse	Create GRN
QC Executive	Add AR Number
QC Head	Approve Material
Example API protection:
@require_role("QC_HEAD")
def approve_material():
This prevents unauthorized operations.
________________________________________
13. Multi-Factor Authentication (MFA)
For additional security, the system may support Google Authenticator MFA.
Flow:
User logs in with username/password
↓
System requests OTP code
↓
User enters Google Authenticator code
↓
Access granted
OTP example:
482915
Libraries:
pyotp
________________________________________
14. Forgot Password Flow
If a user forgets their password:
1️⃣ Click Forgot Password
2️⃣ Enter email address
3️⃣ System sends reset link
Example email:
Click the link below to reset your password:

https://qtrack.com/reset?token=abcd1234
4️⃣ User sets a new password.
________________________________________
15. Password Reset Security
Reset tokens must:
•	expire within 15 minutes
•	be single-use
•	be cryptographically random
Example token:
d1c98e12a9f8f5c12
________________________________________
16. Account Lockout Protection
To prevent brute-force attacks:
Example rule:
5 failed login attempts → lock account for 10 minutes
Example message:
Too many login attempts. Try again later.
________________________________________
17. User Deactivation
If a user leaves the company:
The account is disabled, not deleted.
Field:
is_active = false
Data remains for audit purposes.
________________________________________
18. Session Logout
Logout simply removes the JWT token.
Example:
DELETE /api/auth/logout
Frontend deletes stored token.
________________________________________
19. Audit Logging for Authentication
Every login event should be logged.
Example audit record:
User	Action
raj.wh	Login Success
raj.wh	Password Reset
This helps detect suspicious activity.
________________________________________
20. Security Summary
The authentication system provides:
Closed registration model
Username/password login
Secure password hashing
JWT authentication
Role-based authorization
Email password reset
Optional MFA
Account lockout protection
This ensures the QTrack system remains secure while still simple for users.

Question 19
How will concurrency, transactions, and data consistency be handled in QTrack?
In a warehouse system like QTrack, multiple users may attempt to interact with the same material batch at the same time. For example:
•	two warehouse users issuing stock simultaneously
•	QC updating test results while someone else tries to issue material
•	production requesting material while QC approval is pending
If concurrency is not handled properly, the system can suffer from:
•	negative inventory
•	incorrect stock counts
•	inconsistent batch states
•	corrupted transaction history
Therefore, QTrack must enforce strong transactional integrity using the principles of ACID transactions provided by PostgreSQL.
________________________________________
1. Understanding the Concurrency Problem
Consider the following situation.
Current inventory:
Batch PCM24001
Remaining Quantity = 500 kg
Two warehouse users attempt to issue stock at the same time:
User	Request
User A	Issue 400 kg
User B	Issue 300 kg
If the system does not protect against concurrency:
User A → stock becomes 100 kg
User B → stock becomes -200 kg
This results in negative inventory, which is unacceptable.
________________________________________
2. ACID Transaction Principles
PostgreSQL ensures data integrity through ACID properties.
Property	Meaning
Atomicity	All steps succeed or none are applied
Consistency	Database remains valid
Isolation	Transactions do not interfere
Durability	Committed data is permanent
QTrack will rely heavily on these guarantees.
________________________________________
3. Using Database Transactions
Every inventory update must occur inside a database transaction.
Example flow:
Begin Transaction
↓
Check available stock
↓
Insert stock movement record
↓
Update remaining quantity
↓
Commit Transaction
If any step fails:
Rollback Transaction
This ensures partial operations cannot corrupt data.
________________________________________
4. Example Transaction Implementation
In Python with SQLAlchemy:
with db_session.begin():
    batch = get_batch(batch_id)

    if batch.remaining_quantity < requested_quantity:
        raise Exception("Insufficient stock")

    batch.remaining_quantity -= requested_quantity

    create_stock_movement(...)
This ensures both operations happen together.
________________________________________
5. Row-Level Locking
To prevent multiple users modifying the same batch simultaneously, PostgreSQL supports row-level locks.
When one transaction reads a batch for update:
SELECT * FROM batches
WHERE id = 1
FOR UPDATE;
This locks that row.
Result:
User	Action
User A	acquires lock
User B	must wait
User B cannot modify the batch until User A completes the transaction.
________________________________________
6. Example Concurrency Scenario
Initial state:
Remaining Quantity = 500 kg
User A transaction:
LOCK batch row
Issue 400 kg
Remaining = 100 kg
COMMIT
User B transaction:
Waits for lock
Sees updated quantity (100 kg)
Request = 300 kg
Rejected
This prevents negative inventory.
________________________________________
7. Transaction Isolation Levels
PostgreSQL supports several isolation levels.
Level	Behavior
Read Committed	Default
Repeatable Read	Stronger
Serializable	Strictest
For QTrack, the recommended level is:
READ COMMITTED
It prevents most concurrency issues while maintaining performance.
For critical operations like stock deduction, we can use row-level locks.
________________________________________
8. Preventing Double Processing
Another concurrency issue occurs if a batch is processed twice.
Example:
QC Head approves batch
Warehouse user simultaneously moves batch
Solution:
The backend must validate current status before action.
Example rule:
Material can be issued only if status = APPROVED
If status changes mid-process, the request fails.
________________________________________
9. Optimistic vs Pessimistic Locking
Two concurrency approaches exist.
Optimistic Locking
Assumes conflicts are rare.
Uses version numbers.
Example:
version = 3
If version changes, update fails.
________________________________________
Pessimistic Locking (Recommended Here)
Locks rows immediately.
Example:
SELECT ... FOR UPDATE
Since warehouse operations are critical, pessimistic locking is safer.
________________________________________
10. Handling Status Transitions Safely
Batch status transitions must be controlled.
Example:
QUARANTINE → UNDER_TEST → APPROVED
Invalid transitions must be blocked.
Example rule:
APPROVED cannot move directly back to UNDER_TEST
Status changes must occur through defined workflow endpoints.

Question 30
How should backup, disaster recovery, and long-term system reliability be handled in QTrack?
No production system is complete without a recovery strategy. Even well-built systems can face unexpected failures such as:
•	database corruption
•	server crashes
•	accidental data deletion
•	infrastructure outages
•	human mistakes
Without a backup and recovery plan, all warehouse data could be lost, including:
•	inventory records
•	QC approvals
•	production batches
•	audit logs
Therefore QTrack must implement a structured backup and disaster recovery strategy to guarantee long-term reliability.
________________________________________
1. Backup Strategy Overview
The system should maintain multiple layers of backups.
Recommended model:
Primary Database
        ↓
Daily Automated Backup
        ↓
Weekly Snapshot
        ↓
Long-Term Archive Backup
This ensures that data can be recovered from multiple points in time.
________________________________________
2. Daily Automated Backups
The database should create automatic daily backups.
Example schedule:
Backup Time: 02:00 AM
Retention: 7 days
This means:
•	every day a full snapshot is taken
•	the system keeps backups for the last week
If a failure occurs, the database can be restored to yesterday’s state.
________________________________________
3. Weekly Snapshots
In addition to daily backups, weekly snapshots should be stored.
Example:
Every Sunday
Retention: 4 weeks
These snapshots provide recovery points further in the past.
Example timeline:
Week 1 snapshot
Week 2 snapshot
Week 3 snapshot
Week 4 snapshot
Older snapshots can be automatically removed.
________________________________________
4. Long-Term Archival Backups
For compliance and audit purposes, some backups should be stored long-term.
Example retention:
Monthly archive
Retention: 12 months
These backups protect against rare cases like:
•	long-term unnoticed corruption
•	regulatory investigations
________________________________________
5. Backup Storage Locations
Backups should never be stored on the same server as the main database.
Storage strategy:
Primary Database Server
        ↓
Cloud Backup Storage
In AWS production environments, backups are stored in:
AWS RDS Snapshots
These are automatically replicated across multiple storage systems.
________________________________________
6. Backup Encryption
Backups must be encrypted to prevent data exposure.
Example:
AES-256 encryption
This ensures that even if a backup file is accessed illegally, the data remains unreadable.
________________________________________
7. Backup Verification
Backups must be periodically verified.
Example verification process:
Select random backup
↓
Restore to test database
↓
Check data integrity
This ensures backups are actually usable.
________________________________________
8. Database Recovery Process
If the main database fails, the recovery process should follow these steps.
1. Stop affected services
2. Select latest valid backup
3. Restore database snapshot
4. Restart API server
5. Verify system integrity
This restores the system quickly.
________________________________________
9. Point-in-Time Recovery
PostgreSQL allows point-in-time recovery (PITR).
Example scenario:
Database corrupted at 3:15 PM
Recovery steps:
Restore 2:00 PM snapshot
Replay transaction logs until 3:14 PM
This restores the system just before the failure.
________________________________________
10. Disaster Recovery Plan
A disaster recovery plan prepares for major infrastructure failures.
Examples:
•	cloud region outage
•	server destruction
•	catastrophic data corruption
Recovery approach:
Backup stored in multiple locations
Restore to secondary server
Reconnect application
This minimizes downtime.

The backend API server should also be easily recoverable.
Example strategy:
Infrastructure as Code
Deployment scripts recreate the server automatically.
Example process:
Launch new server
Deploy backend code
Connect to restored database
This rebuilds the application quickly.

For very high reliability, database replication can be used.
Architecture:
Primary Database
        ↓
Replica Database
The replica continuously mirrors the primary database.
If the primary fails:
Replica promoted to primary
This minimizes downtime.

The Senior Backend Design Trick
Use a Domain-Driven Modular Architecture
Instead of building the backend as one big pile of APIs and controllers, you divide the system into independent business domains.
Each domain becomes a self-contained module.
Think of the system like separate mini systems that talk to each other.
________________________________________
Why This Matters for QTrack
Your system already has many business areas:
•	warehouse operations
•	QC
•	QA
•	production
•	finished goods
•	notifications
•	chat
•	authentication
If everything is mixed together, adding features becomes dangerous.
Example of a bad architecture:
controllers/
services/
utils/
models/
Soon everything starts importing everything.
Then one change breaks five features.
